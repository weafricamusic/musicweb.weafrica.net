📋 STEP-BY-STEP CHECKLIST

✅ SETUP (Do Once)

1. Verify Next.js API route exists: /api/push/register
   Location: Your Next.js backend
   Method: POST with Firebase auth verification

2. Verify Supabase table: notification_device_tokens
   Fields: token, user_uid, platform, device_id, country_code, topics (JSONB), last_seen_at

3. Update backend URL in app
   File: lib/features/notifications/services/push_smoke_test_helper.dart
   Change: _backendBaseUrl = 'https://YOUR_NEXTJS_DOMAIN'

4. Initialize FCM in app
   File: lib/services/notification_service.dart
   Called in: main.dart

✅ REGISTRATION FLOW (Per Device)

1. User opens app
2. App logs in with Firebase Auth → gets Firebase UID
3. App requests FCM token → gets device token
4. App calls: POST /api/push/register
   Headers: Authorization: Bearer <Firebase ID Token>
   Body: { token, platform, country, topics }
5. Backend verifies token → saves to Supabase
6. Device is now registered ✅

✅ SENDING FLOW (Each Notification)

1. Admin creates notification in dashboard
2. Admin selects: topic, country, role
3. Admin clicks "Send Now"
4. Backend queries Supabase:
   SELECT fcm_token FROM notification_device_tokens
   WHERE country = 'gh' AND topics @> '["system"]'
5. Backend loops through tokens
6. For each token:
   - Check rate limit
   - Build FCM payload
   - Send to FCM
   - Log result
7. FCM delivers to devices
8. Consumer app receives notification
9. User taps → app routes to screen

✅ RECEIVING FLOW (On Device)

App Open (Foreground):
  → FirebaseMessaging.onMessage fires
  → Show in-app banner
  → Update UI if needed

App Closed (Background):
  → OS shows native notification
  → User taps → app opens
  → FirebaseMessaging.onMessageOpenedApp fires
  → Route to screen based on data.type

⚠️  COMMON MISTAKES (AVOID THESE)

❌ Storing FCM token only in Flutter app
   ✅ Always send to backend and store in Supabase

❌ Sending notifications from consumer app
   ✅ Only backend sends notifications

❌ Hardcoding user IDs in notification logic
   ✅ Query Supabase for matching tokens

❌ Forgetting to check is_active = true
   ✅ Only send to active tokens

❌ Not handling token refresh
   ✅ Listen to onTokenRefresh and update backend

❌ Sending same notification twice
   ✅ Implement rate-limiting

❌ Not logging notification delivery
   ✅ Store in notification_history table

❌ Forgetting to handle different notification types
   ✅ Use data.type for routing logic

🧠 MENTAL MODEL (THE KEY INSIGHT)

Think of it like EMAIL:

Consumer App = Email Client (Outlook, Gmail app)
  - You read emails
  - You don't send emails to everyone
  - You only send to specific people

Backend = Email Server (SMTP)
  - Handles authentication
  - Routes messages
  - Applies rules (spam filters = rate limits)

Supabase = Address Book
  - Stores email addresses (FCM tokens)
  - Stores contacts (users)
  - You query it to find who to send to

Admin Dashboard = Marketing Tool (Mailchimp)
  - You compose the message
  - You select the audience
  - You click "Send"
  - Server does the actual sending

FCM = Email Delivery (SendGrid, Postmark)
  - Takes your message
  - Delivers to inboxes
  - Handles retries if recipient offline

💡 KEY CONCEPTS
===============

Firebase ID Token
  • Proves user is authenticated
  • Required for all API calls
  • Sent in Authorization header

FCM Device Token
  • Unique to this device
  • Used to deliver notifications
  • Stored in Supabase

Device Registration
  • Saves token to database
  • Associates with user account
  • Enables targeting by country/topic

Rate-Limiting
  • Prevents notification spam
  • Default: 5 per user per day
  • Returns 429 when exceeded
